# LLM API 活用知識まとめ

## 1. APIとは

- アプリケーション同士をつなぐ「窓口」
- LLMのAPIとは「ChatGPTやClaudeの頭脳だけを借りる」仕組み
- チャット画面（claude.ai等）を使わず、自分のプログラムからLLMを呼び出せる

---

## 2. チャット画面とAPIの違い

| | チャット画面（claude.ai等） | API |
|---|---|---|
| 操作 | ブラウザ上で手動 | プログラムから自動 |
| 出力確認 | 都度人間が確認・修正できる | 自動処理なので仕組みが必要 |
| 1ターンの精度 | 同じ | 同じ（モデルは変わらない） |
| カスタマイズ | 限定的 | 自由度が高い |
| コスト | 月額固定 | 従量課金 |

**ポイント：1ターンの出力精度はチャット画面とAPIで変わらない。違いは「往復の導線（人間の介在有無）」**

---

## 3. 料金の仕組み

- **従量課金**：使った分だけ支払う（プリペイド方式）
- **トークン**：テキストの単位。インプット（送った文章）とアウトプット（返ってきた文章）両方で課金
- **クレジット**：事前にチャージした残高。なくなったら止まる
- **クレジットの有効期限**：購入から1年で失効
- **自動継続はなし**：残高がなくなったら止まるだけ

### 各社の料金感（2026年時点）
- Claude API：従量課金、モデルによって異なる
- ChatGPT API：従量課金、モデルによって異なる
- Gemini API：無料枠あり、超えたら従量課金

**試す分には$20（約3,000円）あれば数ヶ月〜年単位で使える**

## 4. APIキーの管理

- APIキーはクレカと紐づいた「秘密の鍵」
- **絶対にやってはいけないこと**
  - コードに直接書く
  - GitHubにアップロードする
  - チャット画面に貼り付ける
  - 他人に見せる

### 正しい管理方法

```
# .envファイルに記載（このファイル自体は共有しない）
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxx

# コードでは環境変数から読み込む
import os
api_key = os.environ.get("ANTHROPIC_API_KEY")
```

### プロジェクトごとにAPIキーを分けるべき理由
- どのプロジェクトがどれだけ使っているか把握できる
- キーが漏れたときに該当プロジェクトのキーだけ無効化できる

## 8. system prompt

### system promptとは

会話が始まる前にAIへ渡す「隠れた指示書」。ユーザーには見えない。

```
[system prompt] ← ユーザーには見えない
　　↓
ユーザー: こんにちは
　　↓
Claude:（system promptに従って返答）
```

### 設定できる内容

- **口調・キャラ**：「敬語を使わず友達口調で」「絵文字は使わないで」
- **役割・専門性**：「あなたはExcelの専門家です」
- **出力フォーマット**：「回答は必ず箇条書きで」「Markdownは使わずプレーンテキストで」

### コードでの書き方（Claude API）

```python
response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    system="ここに指示を書く。普通の日本語でOK。",  # ← これだけ追加
    messages=conversation_history
)
```

### 各APIでの書き方の違い

| API | 書き方 |
|---|---|
| Claude（Anthropic） | `system="..."` として専用パラメータで渡す |
| ChatGPT（OpenAI） | `messages`の中に `{"role": "system", "content": "..."}` として入れる |
| Gemini（Google） | `system_instruction="..."` として専用パラメータで渡す |

書き方は微妙に違うが**概念は同じ**。一度書いたsystem promptは他のAPIにもほぼそのまま流用できる。

### 業務での活用イメージ

```
system prompt：「PDFの内容を読み取り、品名・数量・金額を抽出して表形式で返してください」
　　↓
毎回指示しなくても、常にそのフォーマットで返してくれる
```

## 9. Claude Codeでの開発Tips

### ターミナル出力をそのまま共有する方法

Claude Codeのプロンプトで `!` を先頭につけると、コマンドの出力がそのままセッションに流れ込む。

```
! python script.py
! pip list
! python --version
```

コピペ不要で、エラーログなどをそのまま共有できる。

**注意**：`input()` を使う対話型スクリプトは `!` では動かない。その場合は別ターミナルで実行してログをファイルに保存し、`! type log.txt` で共有する。

## 11. APIのレスポンスの仕組み

### APIが返すのは常に「テキスト」だけ

CSVでも会話文でも、APIが返すのはすべてただの文字列。

```python
response = client.messages.create(...)
response.content[0].text  # ← これがすべて。ただの文字列。
```

### 用途別の使い方の違い

```
【chatbot.py】
API → "こんちは～" → print() で表示

【pdf_analyzer.py】
API → "物件名,フロア,...\nメットライフ..." → ファイルに書き込み → .csv
```

受け取り方は同じ。**使い方がちがうだけ。**

### CSVファイルの生成はこちら側の処理

```python
with open("output.csv", "w", encoding="utf-8-sig") as f:
    f.write(result)  # APIから返ってきたテキストをそのまま書き込む
```

「変換処理」は何もしていない。拡張子を `.csv` にしたことでExcelが認識するだけ。`.txt` にしても中身はまったく同じ。

### なぜCSV形式で返してくれたのか

system promptで「CSV形式で出力してください」と指示したから。APIは指示通りのフォーマットのテキストを返しただけ。

### APIの入出力まとめ

| | Claude API |
|---|---|
| 入力できるもの | テキスト・画像・PDF・動画など |
| 出力できるもの | **テキストのみ** |
| 画像を生成する | ❌（DALL-E等の別APIが必要） |
| グラフ・Excelを生成する | ✅（Python側で処理する） |

### system promptで大規模な指示を管理する方法

**パターン1：別ファイルに書いて読み込む（業務推奨）**

```python
with open("prompts/system_prompt.txt", "r", encoding="utf-8") as f:
    system_prompt = f.read()
```

プログラム知識なしで編集できるので、現場担当者に渡しやすい。

**パターン2：コード内に直接書く**

```python
system_prompt = """
あなたはデータ入力の専門家です。
...
"""
```

開発中の試行錯誤向け。

### system と messages の違い

```
system   = AIへの「事前設定」。会話の外側にある。
messages = 実際の「会話のやりとり」。
```

| | system | messages |
|---|---|---|
| 目的 | AIの振る舞いを定義する | AIに考えさせるための材料 |
| 内容 | 役割・ルール・出力形式 | 会話のやりとり |
| 変わるタイミング | 基本変えない | 毎回追加されていく |
| ユーザーに見える？ | 見えない | 見える |

```python
client.messages.create(
    system="不動産の専門家として答えてください",  # 設定。ずっと効き続ける
    messages=[
        {"role": "user",      "content": "渋谷の物件は？"},
        {"role": "assistant", "content": "渋谷には〇〇があります"},
        {"role": "user",      "content": "賃料は？"},  # 今の質問
    ]
)
```

chatbot.pyで会話履歴が引き継がれるのは、`messages`に過去の会話をどんどん追加しているから。`system`はずっと同じ設定として裏で効き続ける。

共通プロンプト（固定・ファイル管理）

## 12. セキュリティ・稟議対応

### APIキー管理（開発者向け）

- APIキー・トークン・パスワード・DB接続文字列はすべて「シークレット」として同様に管理
- GitHubはAPIキーっぽい文字列を自動検出して警告する機能あり
- 会社で使う場合はIT部門のセキュリティポリシーを確認すること

---

### 社内稟議で論点になるポイント

**一番の論点：情報漏洩リスク**

APIに送ったデータは各社のサーバーに送信される。

```
社内の機密情報・個人情報を含むPDF
　　↓ APIに送信
　　↓ Anthropic/OpenAI/Googleのサーバーで処理
　　↓ 結果が返ってくる
```

---

### 稟議で揃える情報

| 確認項目 | 内容 |
|---|---|
| データの学習利用 | 送信したデータがモデル学習に使われるか |
| データ保持期間 | サーバーにどれくらい残るか |
| 第三者提供 | 各社以外に渡るか |
| 準拠規格 | SOC2・ISO27001・GDPRなど |
| 契約形態 | 利用規約のみ or 個別契約可能か |

**証跡として揃えるもの**
- 各社の利用規約・プライバシーポリシー（URL・PDF保存）
- APIの利用規約（チャット画面とは別になっていることが多い）
- Enterpriseなら DPA（データ処理契約書）

---

### プランの違い（個人 vs Enterprise）

```
個人プラン（今使っているやつ）
  → 従量課金、利用規約に同意するだけ

Enterpriseプラン
  → 会社と個別契約、セキュリティ要件を詰められる
```

| | 個人プラン | Enterprise |
|---|---|---|
| 契約形態 | 利用規約に同意するだけ | 会社間で個別契約 |
| データ保持 | 規約通り | ゼロ保持など交渉可能 |
| DPA締結 | ❌ | ✅ |
| SLA（稼働保証） | なし | あり |
| 請求 | クレカ従量 | 請求書払い可能 |

稟議的には「Enterpriseで契約するから情報漏洩リスクはこの条件で担保されている」という説明ができるようになる。

---

### DPA（データ処理契約書）とは

会社とAPI提供会社の間で「あなたのデータをこういうルールで扱います」という約束を書面にしたもの。

- **利用規約**：一方的なAPI会社のルール
- **DPA**：会社とAPI会社が合意した双方向の法的契約

漏洩時に責任追及できる根拠になる。個人情報・機密情報を扱う場合は特に重要。

---

### 各社の現状感（大まかに）

| | Claude | ChatGPT | Gemini |
|---|---|---|---|
| API学習利用 | なし | なし | なし |
| Enterprise契約 | あり | あり | あり |
| DPA締結 | 可能 | 可能 | 可能 |

※プランや契約時期によって変わるので必ず最新の公式ドキュメントで確認。

---

### データの機密レベルによる判断基準

```
小規模・公開情報のみ
　→ 社内ルールを決めるだけでOK

中規模・社外秘あり
　→ Enterpriseプランで利用規約をちゃんと読む

個人情報・機密情報を扱う
　→ DPA締結して責任の所在を明確に
```

**APIに投げてOKなデータの例**
- 公開情報の整理・要約
- 社内の非機密文書の分析

**要検討なデータの例**
- 顧客の氏名・住所・連絡先が入った書類
- 契約書・見積書などの社外秘文書
- 従業員情報

**現実的な社内ルール例（DPAなしでも稟議が通りやすくなる）**
- 個人情報が含まれる文書はAPIに投げない
- 社外秘レベル以上はNG
- 公開情報・社内資料のみOK
